# Azure Node.js Web Application Deployment Report

## Solution Overview

This project demonstrates the deployment and operational management of a Node.js web application hosted on Microsoft Azure App Service. The implementation incorporates cloud-native deployment practices, automated delivery pipelines, application monitoring, configuration management, and scalability planning.

The application is built using the Express.js framework and deployed within a managed Azure Platform-as-a-Service (PaaS) environment, reducing infrastructure management overhead while providing reliability, security, and operational flexibility.

---

# Application Runtime Environment

## Platform Configuration

The application is executed using the following runtime environment:

| Component        | Configuration  |
| ---------------- | -------------- |
| Runtime Platform | Node.js 22 LTS |
| Framework        | Express.js 4.x |
| Operating System | Linux          |

### Technology Selection Rationale

Node.js 22 Long-Term Support (LTS) was selected because it provides a stable and secure execution environment backed by ongoing maintenance and security updates. The release also includes performance improvements, enhanced JavaScript capabilities, and optimizations delivered through the latest V8 engine enhancements.

Linux was chosen as the hosting platform because it offers native support for Node.js workloads, lower resource consumption, faster startup performance, and improved cost efficiency compared to Windows-based hosting environments.

---

# Azure Hosting Configuration

## App Service Plan Selection

### Hosting Tier

| Property          | Value    |
| ----------------- | -------- |
| Pricing Tier      | Basic B1 |
| CPU Allocation    | 1 Core   |
| Memory Allocation | 1.75 GB  |

### Justification for Tier Selection

The Basic B1 plan was selected because it provides dedicated computing resources while remaining cost-effective for development and small production workloads.

Key advantages include:

* Dedicated virtual machine resources.
* Always On functionality.
* Support for SSL certificates.
* Custom domain integration.
* Manual scaling capabilities.
* Integration with Azure monitoring tools.

Unlike the Free tier, which imposes CPU execution limits, the Basic plan provides consistent performance without daily usage restrictions.

# Deployment Automation Strategy

## Continuous Integration and Continuous Delivery

Application deployment is fully automated through GitHub integration with Azure App Service Deployment Center.

### Deployment Workflow

1. Source code is maintained within a GitHub repository.
2. The Azure App Service is connected directly to the repository's main branch.
3. Every code commit automatically triggers a deployment workflow.
4. GitHub Actions executes the build process.
5. Azure Oryx detects the application stack and installs dependencies.
6. The application package is generated and deployed to Azure App Service.

This automated workflow eliminates manual deployment activities and ensures rapid delivery of application updates.

---

## Build and Packaging Process

During deployment, Azure's Oryx build engine performs the following actions:

* Detects the Node.js runtime environment.
* Reads project configuration from package.json.
* Executes dependency installation using npm.
* Packages application artifacts for deployment.
* Publishes the application to Azure App Service.

This process ensures consistency between development and production environments.

# Application Startup Configuration

## Startup Procedure

The platform automatically detects the application's startup command from the package configuration file.

```json
"scripts": {
  "start": "node app.js"
}
```

Because Azure App Service recognizes this standard Node.js startup definition, no additional startup commands or custom scripts are required.

# Error Handling and User Experience

## Custom Not Found Page

A dedicated error handling mechanism has been implemented to improve user experience when invalid URLs are requested.

### Functionality

When a user attempts to access a route that does not exist:

* The request is intercepted by Express middleware.
* An HTTP 404 status code is returned.
* A custom-designed EJS template is rendered.

This approach provides users with a professional error page rather than a generic browser-generated message.

# Environment Configuration Management

## Application Settings

Runtime configuration values are stored securely within Azure App Service Configuration settings instead of being embedded directly in the application source code.

### Configured Variables

| Setting Name | Assigned Value |
| ------------ | -------------- |
| APP_ENV      | production     |
| APP_NAME     | Azure Node App |

### Benefits

* Improved security.
* Easier configuration management.
* Separation of code and environment-specific settings.
* Simplified deployment across multiple environments.

Within the application, these values are accessed using standard Node.js environment variables:

```javascript
process.env.APP_ENV
process.env.APP_NAME
```

# Scalability and Capacity Management

The deployed solution supports both horizontal and vertical growth strategies.

---

## Horizontal Expansion

### Scale-Out Capability

The Basic B1 tier allows administrators to increase the number of application instances from one server to a maximum of three.

Benefits include:

* Increased capacity for concurrent users.
* Improved traffic distribution.
* Better application responsiveness during periods of high demand.

Azure automatically distributes incoming requests across available instances.

---

## Vertical Expansion

### Scale-Up Capability

The hosting plan can be upgraded to larger service tiers such as:

* B2
* B3
* Premium P1v3
* Premium P2v3

These upgrades provide additional CPU and memory resources.

---

## Auto-Scaling Considerations

The Basic B1 plan does not support automatic rule-based scaling.

Features such as:

* CPU-triggered scaling
* Memory-based scaling
* Schedule-based scaling

require upgrading to the Standard (S1) tier or higher.

---

## When Horizontal Scaling Is Appropriate

Scale-out should be considered when:

* User traffic increases significantly.
* CPU utilization remains consistently high.
* Memory usage approaches resource limits.
* Page response times begin to increase.

---

## When Vertical Scaling Is Appropriate

Scale-up is recommended when:

* Additional memory is required.
* CPU-intensive workloads are introduced.
* Advanced App Service capabilities are needed.

Examples include:

* Deployment slots
* Virtual Network integration
* Automated backups
* Advanced networking features

---

# Operational Recommendations

To maintain application performance and availability:

2. Review application logs for recurring issues.
3. Configure Application Insights alerts.
